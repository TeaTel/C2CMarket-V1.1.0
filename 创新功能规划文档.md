## 1. 平台架构转型

### 1.1 功能概述

对项目整体运作逻辑进行系统性重构，实现从传统电商平台向社区论坛模式的战略转型，构建兼具交易与社交属性的综合平台。核心变革在于将平台从"以商品为中心"的交易集市进化为"以人为中心"的社交交易生态——用户不再仅仅是买卖双方，而是社区参与者、内容贡献者和兴趣共建者。

### 1.2 核心目标

| 目标         | 说明                                     |
| ---------- | -------------------------------------- |
| **社交化转型**  | 引入帖子、评论、点赞、关注等社区互动机制，打破当前纯交易的产品边界      |
| **内容驱动交易** | 以用户生成内容（UGC）为流量入口，商品嵌入社区内容流，实现"边逛边买"   |
| **双模式并存**  | 保留原有交易功能完整性的同时叠加社区层，用户可选择纯交易模式或社区沉浸模式  |
| **信息流重构**  | 首页从"商品瀑布流"升级为"兴趣信息流"，融合帖子、商品、活动等多种内容形态 |

### 1.3 主要实现要点

#### 1.3.1 后端架构演进

**新增实体与数据表**（不影响现有 7 张业务表）：

```sql
-- 帖子表（社区内容核心）
CREATE TABLE posts (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    title VARCHAR(200) NOT NULL,           -- 帖子标题
    content TEXT NOT NULL,                  -- 帖子正文（富文本）
    post_type VARCHAR(30) NOT NULL,        -- 类型：DISCUSSION/SHOWCASE/HELP/ACTIVITY
    board_id BIGINT,                        -- 所属板块（兴趣社区）
    view_count INT DEFAULT 0,
    like_count INT DEFAULT 0,
    comment_count INT DEFAULT 0,
    is_pinned TINYINT(1) DEFAULT 0,        -- 是否置顶
    is_essence TINYINT(1) DEFAULT 0,       -- 是否精华
    status VARCHAR(20) DEFAULT 'PUBLISHED',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 帖子评论表
CREATE TABLE post_comments (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    post_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    parent_id BIGINT,                        -- 父评论ID（支持嵌套回复）
    content TEXT NOT NULL,
    like_count INT DEFAULT 0,
    status VARCHAR(20) DEFAULT 'PUBLISHED',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (post_id) REFERENCES posts(id),
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (parent_id) REFERENCES post_comments(id)
);

-- 点赞记录表（通用，可扩展至商品、评论等）
CREATE TABLE user_likes (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    target_type VARCHAR(30) NOT NULL,       -- 点赞目标类型：POST/COMMENT/PRODUCT
    target_id BIGINT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_target (user_id, target_type, target_id)
);

-- 用户关注表
CREATE TABLE user_follows (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    follower_id BIGINT NOT NULL,            -- 关注者
    followee_id BIGINT NOT NULL,            -- 被关注者
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_follow (follower_id, followee_id)
);
```

**新增 API 模块**（与现有 API 路由共存，互不干扰）：

```
/api/v2/feed              → 聚合信息流接口（帖子+商品混合推荐）
/api/v2/posts             → 帖子 CRUD
/api/v2/posts/{id}/comments → 帖子评论管理
/api/v2/likes             → 点赞/取消点赞
/api/v2/follows           → 关注/取消关注
/api/v2/users/{id}/profile → 用户社区主页（含发帖、动态等）
```

**架构调整原则**：

- 采用 `/api/v2/` 版本前缀区分新社区功能 API，确保原有 `/api/*` 全部路由不受影响
- 使用 Spring AOP 切面拦截，对商品浏览量、帖子互动量等指标进行异步统计
- 新增 `FeedService` 聚合服务，负责混合推荐算法的信息流生成
- 复用现有 `SecurityUtils`、`JwtUtil`、`Result`、`ErrorCode` 等基础设施

#### 1.3.2 前端架构演进

**路由扩展**（保留全部现有路由，追加新路由）：


- `FeedList.vue`：混合信息流组件，支持帖子+商品混排展示
- `PostEditor.vue`：富文本帖子编辑器组件
- `CommentSection.vue`：通用评论区域组件（帖子、商品详情复用）
- `UserBadge.vue`：用户徽章/标签展示组件

**导航调整**（保留现有 5 个 TabBar）：

- 首页 Tab 默认展示聚合信息流 `FeedList`
- 在首页顶部添加"交易/社区"视图切换标签
- 新增社区顶部导航栏（各兴趣板块入口）

### 1.4 预期用户价值

- **用户粘性提升**：从"有需求才打开"的纯工具型应用转变为"日常刷帖、发现兴趣"的社交型应用
- **交易转化率提高**：社区内容自然引导用户发现所需商品，降低交易决策门槛
- **信任体系构建**：通过互动历史（发帖质量、评论回复、点赞记录）建立更立体的用户信誉
- **校园文化沉淀**：社区内容沉淀校园生活记忆，形成独特的校园数字资产

## 2. 商品故事功能

### 2.1 功能概述

为商品详情页添加"故事"内容模块，允许用户在发布或编辑商品时为其撰写背景故事、使用体验或情感联结描述（区别于已有的 `description` 纯商品描述字段）。通过叙事性内容赋予商品情感附加值与差异化竞争力，将冰冷的二次交易转化为温暖的物品传承。

### 2.2 核心目标

| 目标         | 说明                             |
| ---------- | ------------------------------ |
| **情感赋能**   | 让每件商品承载独特的故事和记忆，超越简单的物品属性与价格对比 |
| **差异化竞争**  | 相同品类/成色的商品通过故事内容形成品质区隔和情感溢价    |
| **内容二次传播** | 优质故事可被推荐至社区信息流，吸引非购物动机的流量      |
| **交易信任增强** | 卖家通过分享使用经历增强买家对商品真实性和历史信息的信任感  |

### 2.3 主要实现要点

#### 2.3.1 数据模型扩展

**方案**：在现有 `products` 表上追加字段（ALTER TABLE ADD COLUMN，非破坏性变更），保留全部原有字段和索引：

```sql
ALTER TABLE products
ADD COLUMN story_title VARCHAR(200) COMMENT '故事标题',
ADD COLUMN story_content TEXT COMMENT '故事正文（富文本/长文本）',
ADD COLUMN story_images TEXT COMMENT '故事配图URL（JSON数组，与商品主图独立）',
ADD COLUMN has_story TINYINT(1) DEFAULT 0 COMMENT '是否有故事内容（便于筛选）';
```

**设计说明**：

- 与现有 `description` 字段明确区分：`description` 用于客观描述商品参数（型号、尺寸、成色、瑕疵等），`story_content` 用于主观叙事性内容
- `story_images` 独立于 `images`（商品实拍图），允许用户上传使用场景图、纪念照等
- `has_story` 为冗余字段，便于首页/列表快速筛选"有故事的商品"

#### 2.3.2 后端接口扩展

**修改现有接口**（保持向后兼容）：

- `POST /api/products`：请求体新增可选字段 `storyTitle`、`storyContent`、`storyImages`
- `PUT /api/products/{id}`：同上
- `GET /api/products`：响应体新增 `hasStory` 字段，支持查询参数 `hasStory=true` 过滤
- `GET /api/products/{id}`：详情响应包含故事模块完整数据

**新增专属接口**：

```
GET /api/v2/stories/feed              → 故事内容信息流（可在社区首页推荐）
GET /api/v2/stories/user/{userId}     → 某用户的所有故事商品列表
```

#### 2.3.3 前端实现

**修改现有页面**（增量修改）：

`ProductDetailPage.vue`：

- 在商品图片画廊与价格信息区域之后，插入"商品故事"模块
- 故事模块采用杂志式排版：大标题 + 配图轮播 + 正文（支持富文本渲染）
- 若商品无故事内容，该模块不渲染（零感知回退）
- 提供"分享故事"按钮，生成分享卡片（封面+标题+摘要）

`ProductCreatePage.vue` / `ProductEditPage.vue`：

- 在商品描述表单区域之后增加"撰写商品故事（可选）"模块
- 提供故事标题输入框、富文本编辑器（简易 Markdown 或所见即所得）、故事配图上传区
- 界面提示示例文案（如"分享你和这件物品的故事，让交易更有温度"）

**新增组件**：

- `StoryCard.vue`：在信息流中展示的故事卡片
- `StoryEditor.vue`：故事撰写编辑器组件
- `ProductStory.vue`：商品详情中的故事展示模块

### 2.4 预期用户价值

- **买家决策体验升级**：在比较价格和成色之外，通过故事获得情感共鸣，形成"带走的不只是物品，还有一段校园记忆"的认知
- **卖家溢价空间**：优质故事使商品从同质化竞争中脱颖而出，买家愿意为情感附加值支付更高金额
- **内容生态丰富度**：商品故事成为社区内容的重要组成部分，为平台架构转型提供高质量 UGC 供给
- **毕业季情感场景**：大四学生离校时的物品传递故事，天然具有传播性与共情力

## 3. 非货币交易体系

### 3.1 功能概述

在现有实物交易（商品发布→下单→支付）基础上，新增技能与服务交换模块，建立校园内非现金交易生态系统。用户可以发布技能服务（如摄影、编程辅导、乐器教学）或闲置服务时间，通过物品交换服务或其他非货币资产完成交易，促进资源共享与互助氛围形成。

### 3.2 核心目标

| 目标           | 说明                                        |
| ------------ | ----------------------------------------- |
| **交易形式多元化**  | 打破"一手交钱一手交货"的单一模式，引入以物易物、技能互换、服务置换等多种交易形式 |
| **资源利用率最大化** | 激活学生群体的知识/技能/时间等隐性资源，使其成为可交易的"校园货币"       |
| **降低交易门槛**   | 为预算有限的学生提供零现金交易选项，让更多用户参与平台交易             |
| **构建互助生态**   | 以物易物和技能交换天然带有社交属性，强化社区凝聚力                 |

### 3.3 主要实现要点

#### 3.3.1 数据模型设计

```sql
-- 服务/技能发布表
CREATE TABLE services (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    title VARCHAR(200) NOT NULL,             -- 服务标题
    description TEXT NOT NULL,                -- 服务描述
    category_id BIGINT,                       -- 关联分类
    service_type VARCHAR(30) NOT NULL,        -- 服务类型：SKILL/ITEM_EXCHANGE/TIME_BANK
    exchange_for TEXT,                         -- 期望换取的物品/服务描述
    exchange_type VARCHAR(30) NOT NULL,        -- 交换方式：BARTER(以物易物)/SKILL_SWAP(技能互换)/FREE(赠送)/NEGOTIABLE(协商)
    price DECIMAL(10,2),                       -- 等价估值（参考用，非强制收费）
    images TEXT,                               -- 服务展示图（JSON数组）
    duration INT,                              -- 服务时长（分钟）
    location VARCHAR(200),                     -- 服务地点
    status VARCHAR(20) DEFAULT 'ACTIVE',
    view_count INT DEFAULT 0,
    inquiry_count INT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (category_id) REFERENCES categories(id)
);

-- 交换意向/提议表
CREATE TABLE exchange_proposals (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    service_id BIGINT NOT NULL,               -- 目标服务ID
    proposer_id BIGINT NOT NULL,               -- 发起提议者
    provider_id BIGINT NOT NULL,               -- 服务提供者
    proposal_content TEXT NOT NULL,             -- 提议内容（"我用XX物品换你的摄影服务"）
    proposal_type VARCHAR(30) NOT NULL,         -- 提议类型：OFFER_ITEM/OFFER_SKILL/OFFER_COMBO
    status VARCHAR(30) DEFAULT 'PENDING',       -- 状态：PENDING/ACCEPTED/REJECTED/COMPLETED/CANCELLED
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (service_id) REFERENCES services(id),
    FOREIGN KEY (proposer_id) REFERENCES users(id),
    FOREIGN KEY (provider_id) REFERENCES users(id)
);

-- 交换评价记录
CREATE TABLE exchange_reviews (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    proposal_id BIGINT NOT NULL,
    reviewer_id BIGINT NOT NULL,
    target_user_id BIGINT NOT NULL,
    rating INT NOT NULL CHECK (rating >= 1 AND rating <= 5),
    content TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (proposal_id) REFERENCES exchange_proposals(id),
    FOREIGN KEY (reviewer_id) REFERENCES users(id),
    FOREIGN KEY (target_user_id) REFERENCES users(id)
);
```

```
PUT    /api/v2/proposals/{id}/accept    → 接受提议

PUT    /api/v2/proposals/{id}/complete  → 确认交换完成

```

#### 3.3.3 前端实现

### 3.4 预期用户价值

- **校园共享经济闭环**：物品可买卖，技能可交换，闲置时间可变现——平台覆盖校园内所有可交换资源

## 4. 垂直兴趣社区

### 4.1 功能概述

重构商品分类系统，以垂直兴趣板块页面替代传统多级分类树形结构，如"汉服圈"、"机械键盘圈"、"考研圈"、"摄影圈"等主题社区。每个板块成为独立的内容聚合页，融合同类商品展示、相关帖子讨论、活动发布等功能，实现用户兴趣聚合与精准内容匹配。

### 4.2 核心目标

| 目标          | 说明                                       |
| ----------- | ---------------------------------------- |
| **从分类到社区**  | 将干燥的多级分类树转化为有温度、有人气的兴趣板块，从"找东西"进化为"逛圈子"  |
| **精准流量分发**  | 用户按兴趣自然分流至各板块，商品/内容在匹配场景下获得更高曝光转化        |
| **圈层文化建设**  | 每个垂直板块形成独特的圈层文化（汉服圈的穿搭交流、考研圈的互助氛围）       |
| **长尾品类的激活** | 小众兴趣（手办、古着、乐器）在独立板块中获得集中展示，不至于淹没在海量杂货商品中 |

### 4.3 主要实现要点

#### 4.3.1 数据模型扩展

**方案一（推荐）**：复用现有 `categories` 表并扩展，添加板块元数据表：

```sql
-- 扩展 categories 表（ALTER TABLE，保留原有数据）
ALTER TABLE categories
ADD COLUMN is_board TINYINT(1) DEFAULT 0 COMMENT '是否为垂直兴趣板块',
ADD COLUMN icon VARCHAR(500) COMMENT '板块图标URL',
ADD COLUMN cover VARCHAR(500) COMMENT '板块封面图URL',
ADD COLUMN description TEXT COMMENT '板块简介',
ADD COLUMN member_count INT DEFAULT 0 COMMENT '板块成员数',
ADD COLUMN post_count INT DEFAULT 0 COMMENT '板块帖子数',
ADD COLUMN sort_weight INT DEFAULT 0 COMMENT '推荐权重',
```

## 6. 商品拍卖系统

### 6.1 功能概述

为卖家提供商品拍卖功能选项，在发布商品时可在"一口价"和"拍卖"两种模式间选择。拍卖模式下支持设置起拍价、加价幅度及拍卖时长，买家可实时围观拍卖过程并参与竞价，通过竞争机制增加交易的趣味性、互动性和价格发现效率。

### 6.2 核心目标

| 目标         | 说明                              |
| ---------- | ------------------------------- |
| **价格发现机制** | 通过竞价机制让市场供需关系自然决定商品价格，避免定价过高或过低 |
| **交易趣味性**  | 拍卖过程的竞争性和不确定性增强用户参与感和平台粘性       |
| **稀缺商品溢价** | 限量版、热门商品通过拍卖获得更合理的市场定价和更高的成交价   |
| **围观效应**   | 拍卖过程围观人群产生社交互动和二次传播，提升平台活跃度     |

### 6.3 主要实现要点

#### 6.3.1 数据模型设计

```sql
-- 拍卖扩展表（与 products 表一对一关联，仅在拍卖模式商品时存在记录）
CREATE TABLE auctions (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    product_id BIGINT NOT NULL UNIQUE,           -- 关联商品（一对一）
    start_price DECIMAL(10,2) NOT NULL,          -- 起拍价
    bid_increment DECIMAL(10,2) NOT NULL,        -- 每次最低加价幅度
    current_price DECIMAL(10,2) NOT NULL,        -- 当前最高出价
    buy_now_price DECIMAL(10,2),                 -- 秒杀价/一口价（可选，达到即成交）
    deposit DECIMAL(10,2) DEFAULT 0,             -- 保证金/押金
    start_time DATETIME,                         -- 拍卖开始时间
    end_time DATETIME NOT NULL,                   -- 拍卖结束时间
    status VARCHAR(30) DEFAULT 'PENDING',        -- 状态：PENDING(待开始)/ACTIVE(进行中)/ENDED(已结束)/CANCELLED(已取消)
    winner_id BIGINT,                            -- 最高出价者
    view_count INT DEFAULT 0,                    -- 围观人数（指进入详情页的人数）
    bid_count INT DEFAULT 0,                     -- 出价次数
    auto_extend_minutes INT DEFAULT 0,           -- 延时时长（分钟）：结束时如有新出价自动延长（防秒杀）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (product_id) REFERENCES products(id),
    FOREIGN KEY (winner_id) REFERENCES users(id)
);

-- 出价记录表
CREATE TABLE auction_bids (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    auction_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    bid_amount DECIMAL(10,2) NOT NULL,           -- 出价金额
    bid_type VARCHAR(20) DEFAULT 'NORMAL',       -- 出价类型：NORMAL/AUTO_PROXY
    status VARCHAR(20) DEFAULT 'ACTIVE',         -- 状态：ACTIVE/OUTBID(被超越)/WINNING(当前最高)/RETRACTED(撤回)
    bid_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_auction_bid (auction_id, bid_amount DESC),
    FOREIGN KEY (auction_id) REFERENCES auctions(id),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**设计说明**：

- 拍卖扩展表独立于 `products` 表，对非拍卖商品的存取无任何影响
- `products` 表新增 `sale_mode` 字段区分商品销售模式：`FIXED_PRICE`（一口价，默认）/ `AUCTION`（拍卖）
- `auto_extend_minutes` 机制防止"秒杀"行为：在最后 N 分钟内如有新出价，自动延长相应时长
- 保证金机制增加恶意出价成本，维护拍卖严肃性

#### 6.3.2 后端接口设计

```
-- 拍卖管理 ---
POST   /api/v2/auctions                    → 创建拍卖（需传入商品ID+拍卖参数）
GET    /api/v2/auctions                    → 拍卖列表（支持状态筛选：待开始/进行中/已结束）
GET    /api/v2/auctions/{id}               → 拍卖详情（含当前最高出价、出价记录、参与人数）
PUT    /api/v2/auctions/{id}               → 编辑拍卖参数（仅在 PENDING 状态可修改）
PUT    /api/v2/auctions/{id}/cancel         → 取消拍卖

-- 出价管理 ---
POST   /api/v2/auctions/{id}/bids           → 提交出价

GET    /api/v2/auctions/{id}/bids/my        → 我的出价历史

-- 实时推送（WebSocket） ---
STOMP /topic/auction/{auctionId}           → 拍卖实时频道（新出价广播）
STOMP /user/queue/auction/outbid           → 个人被超越通知

```

**定时任务与异步处理**：

- 使用 `@Async` + `CompletableFuture` 处理拍卖结束后的关联操作（创建订单、通知推送）
- 参与 WebSocket 实时广播的用户数量监控，热点拍卖自动扩容推送通道

## 7. 组织化交易功能

### 7.1 功能概述

开发"成立组织"功能模块，允许用户创建或加入交易组织（如"XX学院二手书互助会"、"南区宿舍家居团"等）。组织成员可通过集体力量进行批量采购或集中销售，提升议价能力与交易效率，将个人行为升格为组织化、常态化的集体交易活动。

### 7.2 核心目标

| 目标          | 说明                                  |
| ----------- | ----------------------------------- |
| **规模议价优势**  | 组织成员集合需求，以集体订单与卖家/外部商家谈判，获得批发价或团购折扣 |
| **降低交易摩擦**  | 组织内部交易基于熟人信任，减少商品信息真伪辨别成本和沟通成本      |
| **常态化交易场景** | 组织成为长期存续的交易单元，不同于一次性的团购活动，可持续运营     |
| **校园组织数字化** | 将线下的兴趣社团、班级群、宿舍楼的交易行为数字化至平台，提升平台渗透率 |

## 8. 用户兴趣画像

### 8.1 功能概述

基于用户买卖商品的类型、浏览行为、收藏偏好、搜索关键词、社区互动内容、拍卖参与记录等多维度行为数据，构建用户兴趣画像系统。系统通过定时计算引擎自动分析并生成可视化用户标签，如"摄影大神"、"考研上岸"、"铲屎官"、"数码玩家"、"汉服同袍"、"金牌卖家"等，并通过多维度雷达图和交易行为时间线直观展示用户的兴趣分布与交易风格，增强用户间基于共同兴趣的信任感与社交连接，同时为平台的个性化推荐系统提供核心数据支撑。

### 8.2 核心目标

| 目标 | 说明 |
|------|------|
| **用户标签化** | 通过算法自动生成趣味化、具象化的用户标签，替代枯燥的个人简介文字，让用户在平台上拥有鲜明的"数字身份" |
| **信任体系增强** | 多维度的数据背书（交易记录、兴趣标签、评价分数、信誉等级）替代单一的好评率，构建立体的用户信任画像 |
| **精准推荐** | 基于兴趣画像实现商品/内容/圈子/"可能认识的人"的个性化推荐，提升匹配效率与用户满意度 |
| **社交破冰** | 共同标签（如"你们都是铲屎官"）成为用户间建立连接的自然桥梁，降低校园社交的连接门槛 |
| **平台数据资产** | 用户画像数据成为平台核心数据资产，驱动所有高级功能（推荐算法、广告投放、风控模型）的智能化运作 |
| **用户激励** | 标签和徽章作为平台内的"荣誉勋章"，激励用户持续贡献高质量内容和诚信交易 |

#### 8.3.1 数据模型设计

```sql
    -- 时间统计
    last_active_at DATETIME,                   -- 最后活跃时间
    profile_version INT DEFAULT 1,             -- 画像版本号（每次重算递增）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 用户标签表（存储自动生成的兴趣标签与成就标签）
CREATE TABLE user_tags (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    tag_name VARCHAR(50) NOT NULL,             -- 标签名称："摄影大神"
    tag_type VARCHAR(30) NOT NULL,             -- 标签类型：INTEREST/IDENTITY/ACHIEVEMENT/STYLE
    confidence DECIMAL(3,2) DEFAULT 0,         -- 标签置信度（0-1）
    source VARCHAR(50) NOT NULL,               -- 标签来源：TRADE/BROWSE/COMMUNITY/MANUAL
    weight INT DEFAULT 1,                      -- 标签权重（决定展示优先级）
    is_displayed TINYINT(1) DEFAULT 1,         -- 是否对外展示
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user_tag (user_id, tag_type),
    INDEX idx_tag_name_weight (tag_name, weight DESC),
    UNIQUE KEY uk_user_tag (user_id, tag_name),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 用户行为轨迹表（画像计算的核心数据源，异步采集写入）
CREATE TABLE user_behaviors (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    behavior_type VARCHAR(30) NOT NULL,        -- 行为类型：VIEW/BUY/SELL/FAVORITE/SEARCH/COMMENT/LIKE/SHARE/BID
    target_type VARCHAR(30) NOT NULL,          -- 目标类型：PRODUCT/CATEGORY/BOARD/POST/USER/SERVICE/AUCTION
    target_id BIGINT NOT NULL,                 -- 目标ID
    category_id BIGINT,                        -- 关联分类（便于按品类统计）
    duration INT DEFAULT 0,                    -- 行为持续时间（秒，浏览停留等）
    metadata TEXT,                             -- 扩展元数据（JSON，存储额外上下文信息）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_behavior (user_id, behavior_type, created_at),
    INDEX idx_behavior_time (created_at),
    INDEX idx_category_behavior (category_id, behavior_type),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 用户兴趣维度快照表（定时计算并缓存，避免实时计算压力，支持历史对比）
CREATE TABLE user_interest_snapshots (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    dimension_name VARCHAR(100) NOT NULL,      -- 维度名称（如"数码产品"）
    dimension_weight DECIMAL(5,2) DEFAULT 0,   -- 兴趣权重（百分比，同一用户各维度之和不强制等于100）
    confidence DECIMAL(3,2) DEFAULT 0,         -- 置信度（行为数据量越大越接近1）
    snapshot_date DATE NOT NULL,               -- 快照日期
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_snapshot (user_id, snapshot_date),
    INDEX idx_snapshot_date (snapshot_date),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

#### 8.3.2 画像计算引擎设计

**计算维度**（6 大维度，20+ 子指标）：

| 主维度 | 子指标 | 数据来源 | 计算方式 | 更新频率 |
|--------|--------|----------|----------|----------|
| **交易偏好** | 购买品类分布、出售品类分布、平均客单价、交易频次、交易时段 | `orders`, `products`, `favorites` | 按品类聚合统计，TF-IDF 加权归一化 | 每日凌晨 |
| **浏览兴趣** | 浏览品类分布、浏览时段分布、页面停留时长、搜索关键词频率 | `user_behaviors` | 基于行为频次 × 时长衰减加权（近 30 天权重更高） | 每日凌晨 |
| **社区活跃度** | 发帖数、评论数、点赞数、在线时长、互动对象分布 | `posts`, `post_comments`, `user_likes` | 按互动类型赋权累加（发帖×5 + 评论×2 + 点赞×1） | 每日凌晨 |
| **信誉评估** | 评价分数、交易完成率、纠纷率、平均响应速度、退款率 | `exchange_reviews`, `orders`, `chat_messages` | 加权综合评分算法——好评率×0.4 + 完成率×0.3 + 响应速度×0.2 + 纠纷率×(-0.1) | 每日凌晨 |
| **身份特征** | 注册时长、认证状态、组织归属、板块成员身份、角色 | `users`, `org_members`, `board_members` | 直接映射为身份标签（如"XX学院"、"XX组织会长"） | 实时/增量 |
| **交易风格** | 价格偏好区间、议价意愿、决策速度、拍卖活跃度、团购参与度 | `auction_bids`, `orders`, `exchange_proposals`, `group_buy_participants` | 统计建模 + K-Means 聚类分析（3 类：理性型/冲动型/探索型） | 每周 |

### 8.4 预期用户价值

| **功能膨胀导致项目失控** | 🔴 高 | 八项功能并行开发，需求蔓延、资源分散、各功能完成度低 | 严格执行四阶段实施路线，阶段间硬性评审；每阶段功能必须达到"可演示"状态才进入下一阶段；设立"需求冻结"机制 |
| **社区转型用户接受度低** | 🔴 高 | 用户习惯纯交易模式，社区化改造后不适应，导致 DAU 下降 | 双模式并存——保留纯交易视图作为"经典模式"；新功能通过引导教程逐步推广；灰度期间密切监控 DAU 变化 |
| **冷启动难题** | 🟡 中 | 社区内容（帖子/评论）初期匮乏，用户进入后看到空白社区即离开 | 运营团队预填充 50+ 篇高质量种子帖子；邀请 20 名校园 KOL 作为首批社区贡献者；初期对发帖用户给予积分/权益激励 |

#### A.7.2 技术风险

| 风险项 | 风险等级 | 描述 | 应对策略 |
|--------|----------|------|----------|
| **WebSocket 高并发压力** | 🔴 高 | 拍卖实时出价和社区消息推送的 WebSocket 连接数过大，服务端压力激增 | 实施 WebSocket 连接数限流（单用户最多 10 个订阅）；热点拍卖采用独立 WebSocket 节点隔离；引入 Redis Pub/Sub 实现多节点广播 |
| **数据库写入瓶颈** | 🔴 高 | `user_behaviors` 表日均写入量可能达到百万级，单表性能瓶颈 | 实施分区表策略（按月份分区）；引入时序数据库（如 TDengine）存储行为数据；冷数据按月归档到对象存储 |
| **画像计算资源消耗** | 🟡 中 | 每日凌晨全量用户画像重算消耗大量 CPU 和内存资源 | 增量计算——仅重算近 7 天有活跃行为的用户；计算任务分批串行执行避免资源争抢；预留独立计算节点 |
| **前后端接口不兼容** | 🟡 中 | 多阶段迭代开发过程中 V2 API 接口频繁变更，前端联调成本高 | 采用 API First 设计——先定义 OpenAPI 3.0 规范文档，前后端基于契约并行开发；使用 Mock Server 模拟未就绪接口 |
| **第三方依赖风险** | 🟢 低 | 引入 ECharts、markdown-it 等新依赖可能带来安全漏洞或版本冲突 | 使用 Dependabot 自动监控依赖安全更新；在 CI 流水线中集成 Trivy 漏洞扫描；锁定依赖的次版本号 |
| **数据一致性风险** | 🟡 中 | 团购参与人数、拍卖当前价格等并发更新的数据可能出现不一致 | 使用乐观锁（version 字段）处理团购参与计数；拍卖出价使用数据库行锁（SELECT ... FOR UPDATE）保证价格递增 |

#### A.7.3 运营风险

| 风险项 | 风险等级 | 描述 | 应对策略 |
|--------|----------|------|----------|
| **社区内容质量失控** | 🟡 中 | 开放社区发帖后可能出现低质量内容、垃圾广告、违规信息 | 预置敏感词过滤（基于 DFA 算法）；引入社区举报机制；初期设置发帖频率限制（新用户每小时最多 3 帖） |
| **拍卖恶意行为** | 🟡 中 | 卖家使用小号抬价、买家恶意出价不付款 | 保证金机制（参与拍卖需缴纳商品起拍价 5% 的保证金）；关联实名认证限制小号注册；恶意行为记录并限制参与后续拍卖 |
| **团购履约纠纷** | 🟡 中 | 团购发起人收款后不发货、商品与描述不符 | 团购资金平台托管（引入微信支付/支付宝担保交易）；团购发起人需完成实名认证 + 缴纳保证金；建立团购纠纷快速仲裁通道 |
| **用户隐私合规** | 🟡 中 | 用户行为数据采集和画像展示可能涉及个人信息保护合规问题 | 行为数据采集遵循最小必要原则；提供行为数据采集开关（用户可自主关闭）；隐私政策和法律条款持续更新；定期合规审查 |

#### A.7.4 风险等级汇总

| 风险等级 | 数量 | 典型风险项 |
|----------|------|-----------|
| 🔴 高风险 | 5 | 功能膨胀、用户接受度、行为采集性能、WebSocket 并发、DB 写入瓶颈 |
| 🟡 中风险 | 13 | 冷启动、画像耗时、冷启动推荐、标签准确性、隐私顾虑、数据膨胀、组织冷启动等 |
| 🟢 低风险 | 4 | 推荐信息茧房、组织数据泄露、第三方依赖、组织商品泄露 |
